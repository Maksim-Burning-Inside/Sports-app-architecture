# Архитектурный концепт спортивной экосистемы
## Контур 11: Матрица архитектурных решений (ADR Matrix)

Данный документ содержит реестр ключевых архитектурных решений, определяющих технологический фундамент платформы на втором этапе проектирования. Матрица фиксирует суть каждого выбора, его последствия для системы, сопутствующие инженерные риски и отвергнутые технологические альтернативы.

---

### Реестр стратегических решений (Architectural Decision Records)

| № | Название решения | Суть подхода | Последствия (Плюсы) | Инженерные риски (Минусы) | Отвергнутые альтернативы и причины отказа |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **ADR-01** | Event-Driven архитектура как основа взаимодействия | Все прикладные домены общаются между собой асинхронно через шину событий при фиксации тренировок, расчете износа и начислении наград. | • Высокая устойчивость доменов при пиковых нагрузках соревнований.<br>• Независимость работы доменов друг от друга.<br>• Параллельная обработка одного события разными сервисами. | • Повышенная сложность отладки сквозных распределенных потоков.<br>• Необходимость жесткого контроля схем сообщений.<br>• Данные обновляются с небольшой задержкой. | **Синхронный REST HTTP / gRPC (межсервисный)**<br>Прямые связанные вызовы между доменами привели бы к каскадным падениям доменов, для столь большой системы это было бы недопутсимо. |
| **ADR-02** | Стратегия полиглотного хранения данных | Отказ от одной универсальной СУБД в пользу трех специализированных решений под конкретные типы бизнес-задач. | • Выборка напарников поблизости без тяжелых JOIN-запросов.<br>• Максимальное сжатие Big Data телеметрии на дисках.<br>• Транзакционная точность при оформлении финансовых заказов. | • Усложнение общей инфраструктуры хранения данных.<br>• Необходимость поддержки трех разных типов баз командами эксплуатации.<br>• Сложность сквозного резервного копирования. | **Единая реляционная СУБД (MS SQL / Oracle)**<br>Хранимые массивы данных отличаются по своему назначению, социальный домен и домент телеметрии работали бы значительно хуже без специализированных хранилищ под их задачи, а общий огромный поток данных перегужал бы базы. |
| **ADR-03** | Multi-Cloud развертывание без vendor lock-in | Использование независимых open-source инструментов для развертывания модулей у разных облачных провайдеров под региональные законы. | • Свобода выбора облачной площадки под законы конкретной страны (GDPR, ФЗ-152).<br>• Защита бизнеса от внезапного ценового шока со стороны одного вендора.<br>• Гибкое масштабирование. | • Высокие требования к квалификации DevOps-инженеров.<br>• Необходимость создания единой независимой системы мониторинга и сборки кода. | **Proprietary-сервисы провайдеров (AWS Lambda / Azure Functions)**<br>Облачные функции привязывают код к одному вендору, делая невозможным быстрый перенос системы в дата-центры других регионов. |
| **ADR-04** | Децентрализация шлюзов управления трафиком | Разделение входящего сетевого периметра на три изолированных специализированных API-шлюза по типу протоколов. | • Изоляция тяжелого партнерского трафика от розничных пользователей.<br>• Способность удерживать миллионы легких соединений от фитнес-часов одновременно. | • Усложнение настройки сетевых экранов (Firewalls) на периметре.<br>• Необходимость сквозной маршрутизации сквозных идентификаторов запросов. | **Единый универсальный шлюз (Nginx / Envoy в базовой конфигурации)**<br>Один шлюз не справился бы одновременно с миллионами MQTT-пакетов телеметрии и тяжелыми gRPC-файлами остатков, создав сетевую пробку. |
| **ADR-05** | Контекстное использование ИИ без выделенной базы | Интеграция ИИ-агента с существующими базами данных через математические векторные модули расширения. | • Отсутствие затрат на покупку и администрирование отдельной тяжелой базы.<br>• Мгновенный доступ ИИ к актуальным профилям КБЖУ и логам тренировок. | • Повышенная нагрузка на процессоры реляционной СУБД при сложных расчетах планов.<br>• Необходимость оптимизации векторов в общей базе. | **Выделенная векторная СУБД (Pinecone / Milvus)**<br>Развертывание отдельной базы под ИИ нарушает принцип прагматизма, удваивает затраты на сервера и создает хаос при синхронизации данных пользователей. |
