# Архитектурный концепт спортивной экосистемы
## Контур 12: Описание сценариев использования приложения (Use Cases)

Данный документ содержит детальные спецификации трех критических сценариев использования платформы. Каждое описание определяет участников процесса, обязательные условия выполнения, основной пошаговый алгоритм взаимодействия мобильного приложения с внутренними сервисами и альтернативные сценарии обработки событий.

---

### 1. Сценарий использования UC-01: Запись активности в режиме реального времени и офлайн-синхронизация

Процесс регламентирует гарантированный сбор физических метрик пользователя с фитнес-устройств и асинхронную доставку данных на сервер.

* **1.1. Участник процесса.** Авторизованный пользователь приложения.
* **1.2. Предусловие.** Мобильное приложение запущено на смартфоне, фитнес-устройство успешно подключено по беспроводному протоколу ближнего радиуса действия.
* **1.3. Основной сценарий (Интернет-соединение стабильно).**
  1. Пользователь нажимает кнопку запуска тренировки в интерфейсе мобильного приложения.
  2. Приложение открывает постоянную сетевую сессию со шлюзом IoT Ingestion по протоколу MQTT.
  3. Фитнес-устройство каждую секунду передает пакеты физических метрик и географических координат в приложение, а оно перенаправляет их на шлюз в реальном времени.
  4. Шлюз IoT Ingestion принимает потоковые данные, выполняет проверку сессии и асинхронно передает пакеты в Time-Series базу данных для хранения.
  5. Пользователь нажимает кнопку завершения активности, сетевая сессия закрывается, тренировка отображается в дневнике.
* **1.4. Альтернативный сценарий (Сеть отсутствует — Offline-First).**
  1а. На шаге 2 приложение обнаруживает отсутствие интернет-соединения и переводит контур записи в автономный режим.
  2а. Приложение начинает непрерывно записывать секундные пакеты телеметрии с фитнес-устройства во внутреннюю локальную базу данных SQLite на смартфоне.
  3а. При восстановлении сетевого соединения агент синхронизации автоматически упаковывает накопленные в SQLite логи в единый архив с уникальным идентификатором UUID.
  4а. Архив асинхронно отправляется на шлюз IoT Ingestion в фоновом режиме, не блокируя работу интерфейса для пользователя.
  5а. Система распаковывает пакет, сохраняет данные в Time-Series базу данных и восстанавливает цепочку шагов основного сценария.

### 2. Сценарий использования UC-02: Поиск напарников для совместной тренировки поблизости

Процесс отвечает за мгновенное взаимное обнаружение участников сообщества со схожими спортивными целями в одной географической локации.

* **2.1. Участник процесса.** Авторизованный пользователь приложения, готовый к началу активности.
* **2.2. Предусловие.** В настройках смартфона включена геолокация, мобильное приложение имеет стабильный доступ к интернету.
* **2.3. Основной сценарий.**
  1. Пользователь открывает карту в приложении и нажимает кнопку запуска поиска напарников.
  2. Мобильное приложение считывает текущие координаты смартфона и отправляет запрос на шлюз Edge B2C.
  3. Шлюз перенаправляет запрос во внутренний контур домена социализации, который временно заносит идентификатор пользователя и его координаты в распределенную кэш-базу активного поиска.
  4. Сервис гео-сопоставления делает мгновенную выборку из кэш-базы всех участников в радиусе одного километра.
  5. Система одновременно опрашивает Графовой СУБД для сопоставления выбранных профилей по уровню спортивной подготовки и общим целям.
  6. Приложение формирует ротированный по убыванию степени соответствия список людей и отображает их профили на экране пользователя менее чем за одну секунду.

### 3. Сценарий использования UC-03: Расчет амортизации инвентаря и выдача контекстных предложений маркетплейса

Процесс обеспечивает автоматическое определение износа экипировки на основе телеметрии и генерацию таргетированных коммерческих предложений.

* **3.1. Участник процесса.** Внутренние сервисы платформы (автоматический сценарий без прямого участия человека в момент выполнения).
* **3.2. Предусловие.** Пользователь завершил тренировку, данные телеметрии успешно сохранены в Time-Series базе данных.
* **3.3. Основной сценарий.**
  1. Домен телеметрии фиксирует успешную запись тренировки и отправляет асинхронное сообщение с агрегированными показателями эксплуатации в шину событий.
  2. Домен коммерции принимает событие из шины и обновляет цифровой профиль инвентаря в реляционной базе данных, прибавляя новые показатели к общему уровню износа экипировки.
  3. При достижении критических значений износа запускается маркетинговый модуль коммерческого домена.
  4. Модуль проверяет текущую геолокацию пользователя, региональный календарь праздников и запрашивает актуальные остатки нужных моделей оборудования на складах партнеров через шлюз B2B Partner.
  5. Система формирует персональное предложение со скидкой и нативно подмешивает рекламный баннер о замене инвентаря в социальную ленту пользователя.
  6. Одновременно домен геймификации ловит это же событие и добавляет временный промокод на покупку нужного оборудования к списку игровых наград пользователя.

### 4. Сценарий использования UC-04: Скроллинг и динамическое обновление социальной ленты

Сценарий описывает чтение новостной ленты с нативным подмешиванием региональной рекламы без задержек в интерфейсе.

* **4.1. Участник процесса.** Авторизованный пользователь приложения.
* **4.2. Предусловие.** Мобильное приложение подключено к сети через шлюз Edge B2C, у пользователя есть настроенные подписки на других участников или группы.
* **4.3. Основной сценарий.**
  1. Пользователь открывает главный экран приложения и выполняет жест обновления социальной ленты.
  2. Приложение отправляет пакетный запрос на шлюз Edge B2C с указанием текущей страницы (пагинации) и временной метки.
  3. Сервис новостной ленты запрашивает у Графовой СУБД структуру подписок пользователя, чтобы определить список авторов.
  4. Система извлекает предсформированные публикации друзей и групп из кэша оперативной памяти, а домен коммерции параллельно добавляет в этот массив актуальный рекламный баннер на основе региональных акций.
  5. Сформированная лента передается обратно в приложение в течение 500 миллисекунд.
* **4.4. Альтернативный сценарий (Сбой кэша).**
  1а. Если предсформированная лента отсутствует в быстром кэше памяти, сервис осуществляет прямой запрос к базе данных.
  2а. Система собирает публикации по цепочке ID, обновляет кэш и отдает результат пользователю с незначительной задержкой.

### 5. Сценарий использования UC-05: Вступление в социальные спортивные группы и сообщества

Процесс регламентирует создание связи между профилем пользователя и спортивным клубом в графе данных.

* **5.1. Участник процесса.** Авторизованный пользователь приложения.
* **5.2. Предусловие.** Пользователь нашел интересующую его открытую или закрытую группу через строку поиска социальной сети.
* **5.3. Основной сценарий.**
  1. Пользователь нажимает кнопку «Вступить в группу» в карточке спортивного сообщества.
  2. Приложение отправляет запрос на шлюз Edge B2C, который пересылает его в сервис управления связями.
  3. Сервис проверяет статус группы (открытая) и создает новое направленное ребро связи (тип «Участник») между узлом пользователя и узлом группы в Графовой СУБД.
  4. После фиксации связи система генерирует событие в шину данных для автоматической подписки пользователя на будущие публикации этого сообщества.
  5. Интерфейс приложения обновляется, подтверждая успешное вступление.
* **5.4. Альтернативный сценарий (Закрытая группа).**
  1а. Если группа имеет статус закрытой, на шаге 3 система не создает прямую связь, а формирует узел заявки на вступление.
  2а. Администратор группы получает уведомление и одобряет заявку, после чего связь в Графовой СУБД фиксируется по основному сценарию.

### 6. Сценарий использования UC-06: Запрос персональных рекомендаций у индивидуального ИИ-агента

Сценарий описывает генерацию планов питания и тренировок на основе накопленной истории активности.

* **6.1. Участник процесса.** Авторизованный пользователь приложения.
* **6.2. Предусловие.** В базах данных накоплена история тренировок пользователя минимум за две недели, зафиксированы параметры веса и КБЖУ.
* **6.3. Основной сценарий.**
  1. Пользователь заходит во вкладку цифрового наставника и нажимает кнопку обновления тренировочного плана.
  2. Запрос через шлюз Edge B2C поступает в домен геймификации и интеллекта.
  3. Вычислительный сервис ИИ-агента считывает историю пульса и темпа из Time-Series базы данных, а параметры здоровья — из реляционной СУБД.
  4. Модуль расширения преобразует эти метрики в математические векторы и сопоставляет их с эталонными методиками.
  5. ИИ-агент генерирует персональное расписание нагрузок и КБЖУ на следующую неделю, сохраняет его в профиль и выводит на экран пользователя.
* **6.4. Альтернативный сценарий (Недостаточно данных).**
  1а. Если в Time-Series базе данных нет записей за последние две недели, ИИ-агент прерывает расчет планов.
  2а. Система выводит пользователю интерактивную анкету для ручного ввода текущих физических параметров для составления базового расписания.

### 7. Сценарий использования UC-07: Оформление заказа и покупка товаров во внутреннем маркетплейсе

Процесс обеспечивает транзакционную точность покупки инвентаря со списанием бонусов.

* **7.1. Участник процесса.** Авторизованный пользователь приложения.
* **7.2. Предусловие.** Инвентарь добавлен в корзину покупок, на балансе пользователя есть начисленная локальная игровая валюта.
* **7.3. Основной сценарий.**
  1. Пользователь переходит в корзину маркетплейса, выбирает опцию оплаты части заказа баллами и нажимает кнопку подтверждения.
  2. Приложение отправляет транзакционный запрос в домен коммерции через шлюз Edge B2C.
  3. Сервис проверяет актуальность цен и наличие остатков товара на региональном складе партнера через шлюз B2B Partner.
  4. В рамках единой транзакции внутри реляционной СУБД система списывает монеты с баланса геймификации, фиксирует оплату деньгами и переводит статус товара в режим резервирования.
  5. Пользователь получает электронный чек, а домен коммерции отправляет событие в шину данных для обновления логистики дистрибьютора.
* **7.4. Альтернативный сценарий (Товар закончился).**
  1а. Если на шаге 3 партнерский склад через gRPC сообщает об отсутствии товара, транзакция полностью отменяется.
  2а. Деньги и баллы не списываются, а приложение выводит пользователю уведомление с предложением выбрать аналогичный инвентарь.

### 8. Сценарий использования UC-08: Регистрация и участие в глобальных онлайн-соревнованиях

Сценарий регламентирует вход в турнирную сетку и обновление таблиц лидеров при финише.

* **8.1. Участник процесса.** Авторизованный пользователь приложения.
* **8.2. Предусловие.** В системе объявлен открытый международный челлендж или онлайн-марафон с призовым фондом в игровой валюте.
* **8.3. Основной сценарий.**
  1. Пользователь заходит во вкладку соревнований и нажимает кнопку участия в марафоне.
  2. Сервис геймификации добавляет идентификатор пользователя в турнирную таблицу этого события внутри реляционной базы данных.
  3. Пользователь выполняет беговую или силовую активность, телеметрия которой пишется по сценарию UC-01.
  4. При фиксации финиша домен телеметрии выбрасывает событие в шину. Сервис геймификации подхватывает его, рассчитывает итоговое время и обновляет позицию пользователя в таблице лидеров.
  5. Пользователю начисляются призовые монеты и трофеи, а обновленная таблица лидеров кэшируется в оперативной памяти для мгновенного просмотра другими участниками.
* **8.4. Альтернативный сценарий (Дисквалификация).**
  1а. Если при проверке трека в домене телеметрии выявляются технические аномалии, событие финиша помечается как подозрительное.
  2а. Результат не идет в общую таблицу лидеров, а отправляется на проверку модераторам во избежание накруток валюты.
