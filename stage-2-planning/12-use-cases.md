# Артефакт 12. Описание сценариев использования приложения.

---

### 1. Запись активности в режиме реального времени и офлайн-синхронизация.

Процесс регламентирует гарантированный сбор физических метрик пользователя с фитнес-устройств и асинхронную доставку данных на сервер.

* **1.1. Участник.** Авторизованный пользователь приложения.
* **1.2. Окружение.** Мобильное приложение запущено на смартфоне, фитнес-устройство успешно подключено по Bluetooth к телефону, приложение имеет доступ к собираемым данным.
* **1.3. Основной сценарий (Интернет-соединение стабильно).**
  1. Пользователь нажимает кнопку начала тренировки в мобильном приложении.
  2. Приложение открывает постоянную сетевую сессию со шлюзом IoT Ingestion.
  3. Фитнес-устройство каждую секунду передает пакеты физических метрик и гео-данных на Шлюз IoT Ingestion.
  4. Шлюз IoT Ingestion принимает данные, асинхронно передает пакеты в Time-Series базу данных для хранения.
  5. Пользователь нажимает кнопку завершения тренировки, тренировка отображается в дневнике в течение пяти секунд.
* **1.4. Альтернативный сценарий (Сеть отсутствует).**
  1. Пользователь нажимает кнопку начала тренировки в мобильном приложении.
  2. Приложение обнаруживает отсутствие интернет-соединения и переводит запись данных в автономный режим.
  3. Приложение начинает непрерывно записывать пакеты телеметрии с фитнес-устройства во внутреннюю локальную базу данных SQLite.
  4. Пользователь нажимает кнопку завершения тренировки, тренировка отображается в дневнике, временно отображая данные из локальной базы (с ограничениями).
  5. При восстановлении сетевого соединения агент синхронизации автоматически упаковывает накопленные в SQLite логи в единый архив с уникальным идентификатором UUID.
  6. Архив асинхронно отправляется на шлюз IoT Ingestion в фоновом режиме, не блокируя работу интерфейса для пользователя.
  7. Система распаковывает пакет, сохраняет данные в Time-Series базу данных.

### 2. Поиск напарников для совместной тренировки поблизости.

Процесс отвечает за мгновенное взаимное обнаружение участников сообщества со схожими спортивными целями в одной географической локации.

* **2.1. Участник.** Авторизованный пользователь приложения.
* **2.2. Окружение.** В настройках смартфона включена геолокация, мобильное приложение имеет стабильный доступ к интернету.
* **2.3. Основной сценарий.**
  1. Пользователь открывает карту в приложении и нажимает кнопку запуска поиска напарников.
  2. Мобильное приложение считывает текущие координаты смартфона и отправляет запрос на шлюз Edge B2C.
  3. Шлюз перенаправляет запрос во внутренний контур домена социализации, который временно заносит идентификатор пользователя и его координаты в распределенную кэш-базу активного поиска.
  4. Сервис гео-сопоставления делает мгновенную выборку из кэш-базы всех участников в указанном пользователем радиусе.
  5. Система одновременно опрашивает графовую СУБД для сопоставления выбранных профилей по уровню спортивной подготовки и общим целям.
  6. Приложение формирует ротированный по убыванию степени соответствия список людей менее чем за одну секунду и отображает их пользователю.

### 3. Расчет амортизации инвентаря и выдача контекстных предложений маркетплейса.

Процесс обеспечивает автоматическое определение износа экипировки на основе телеметрии и формирование индивидуальных предложений в маркетплейсе.

* **3.1. Участник.** Домен коммерции.
* **3.2. Окружение.** Пользователь завершил тренировку, данные телеметрии успешно сохранены в Time-Series базе данных.
* **3.3. Основной сценарий.**
  1. После окончания тренировки домен телеметрии отправляет асинхронное сообщение с агрегированными показателями эксплуатации в шину событий.
  2. Домен коммерции принимает событие из шины и обновляет цифровой профиль инвентаря в реляционной базе данных.
  3. При достижении критических значений износа запускается процесс формирования индивидуального предложения.
  4. Модуль проверяет текущую геолокацию пользователя, региональный календарь праздников и запрашивает актуальные остатки нужных моделей оборудования на складах партнеров через шлюз B2B Partner.
  5. Система формирует персональное предложение и нативно подмешивает рекламный баннер в социальную ленту пользователя.
  6. Домен геймификации получает событие формирования уникального предложения и "случайно" подмешивает в список ближайших наград промокод именно на нужный товар.

### 4. Скроллинг и динамическое обновление социальной ленты.

Сценарий описывает чтение новостной ленты с нативным подмешиванием региональной рекламы без задержек в интерфейсе.

* **4.1. Участник.** Авторизованный пользователь приложения.
* **4.2. Окружение.** Мобильное приложение подключено к сети через шлюз Edge B2C, у пользователя есть настроенные подписки на других участников или группы.
* **4.3. Основной сценарий.**
  1. Пользователь открывает главный экран приложения и активирует событие обновления ленты (провел пальцем сверху-вниз).
  2. Приложение отправляет пакетный запрос на шлюз Edge B2C с указанием текущей страницы и временной метки.
  3. Сервис новостной ленты запрашивает у графовой СУБД структуру подписок пользователя.
  4. Система извлекает предсформированные публикации друзей и групп из кэша оперативной памяти (исключая те, что уже находились в ленте ранее), а домен коммерции параллельно добавляет в этот массив актуальный рекламный баннер на основе региональных акций.
  5. Сформированная лента передается обратно в приложение в течение 500 миллисекунд.
* **4.4. Альтернативный сценарий (Сбой кэша).**
  1. Если предсформированная лента отсутствует в быстром кэше памяти (первый запуск, вход с нового устройства, вход после долгого отсутствия), сервис осуществляет прямой запрос к базе данных.
  2. Система собирает публикации, обновляет кэш и отдает результат пользователю с незначительной задержкой.

### 5. Вступление в социальные спортивные группы и сообщества.

Процесс регламентирует создание связи между профилем пользователя и спортивным клубом в графе данных.

* **5.1. Участник.** Авторизованный пользователь приложения.
* **5.2. Окружение.** Пользователь нашел интересующую его открытую или закрытую группу через строку поиска социальной сети.
* **5.3. Основной сценарий.**
  1. Пользователь нажимает кнопку «Вступить в группу» на странице спортивного сообщества.
  2. Приложение отправляет запрос на шлюз Edge B2C, который пересылает его в сервис управления связями.
  3. Сервис проверяет статус группы (открытая) и создает новую связь типа «участник» между узлом пользователя и узлом группы в графовой СУБД.
  4. После фиксации связи система генерирует событие в шину данных для автоматической подписки пользователя на будущие публикации этого сообщества.
  5. Интерфейс приложения обновляется, подтверждая успешное вступление.
* **5.4. Альтернативный сценарий (Закрытая группа).**
  1. Если группа имеет статус закрытой, на шаге 3 система не создает прямую связь, а формирует узел заявки на вступление.
  2. Администратор группы получает уведомление и одобряет заявку, после чего связь в графовой СУБД фиксируется по основному сценарию.

### 6. Запрос персональных рекомендаций у индивидуального ИИ-агента.

Сценарий описывает генерацию планов питания и тренировок на основе накопленной истории активности.

* **6.1. Участник.** Авторизованный пользователь приложения.
* **6.2. Окружение.** В базах данных накоплена история тренировок пользователя минимум за две недели.
* **6.3. Основной сценарий.**
  1. Пользователь заходит во вкладку цифрового наставника и нажимает кнопку обновления тренировочного плана.
  2. Запрос через шлюз Edge B2C поступает в домен геймификации.
  3. Вычислительный сервис ИИ-агента считывает историю пульса и темпа из Time-Series базы данных, параметры здоровья из реляционной СУБД.
  4. Специализированный модуль преобразует эти метрики в математические векторы.
  5. ИИ-агент учитывает параметры человека, его запрос и прогнозирует, что ему нужно сделать для достижения результата.
* **6.4. Альтернативный сценарий (Недостаточно данных).**
  1. Если в Time-Series базе данных нет записей за последние две недели, ИИ-агент прерывает расчет планов.
  2. Система выводит пользователю интерактивную анкету для ручного ввода текущих физических параметров, после чего продолжается основной сценарий.

### 7. Оформление заказа и покупка товаров во внутреннем маркетплейсе.

Процесс обеспечивает транзакционную точность покупки инвентаря со списанием бонусов.

* **7.1. Участник.** Авторизованный пользователь приложения.
* **7.2. Окружение.** Инвентарь добавлен в корзину покупок, на балансе пользователя есть начисленная локальная валюта.
* **7.3. Основной сценарий.**
  1. Пользователь переходит в корзину маркетплейса, выбирает опцию оплаты части заказа баллами и нажимает кнопку подтверждения.
  2. Приложение отправляет транзакционный запрос в домен коммерции через шлюз Edge B2C.
  3. Сервис проверяет актуальность цен и наличие остатков товара на региональном складе партнера через шлюз B2B Partner.
  4. В рамках единой транзакции внутри реляционной СУБД система списывает монеты из домена геймификации, фиксирует оплату деньгами и переводит статус товара в режим резервирования.
  5. Пользователь получает электронный чек, а домен коммерции отправляет событие в шину данных для обновления логистики дистрибьютора.
* **7.4. Альтернативный сценарий (Товар закончился).**
  1. Если на шаге 3 партнерский склад через gRPC сообщает об отсутствии товара, транзакция полностью отменяется.
  2. Деньги и баллы не списываются, а приложение выводит пользователю уведомление с предложением выбрать аналогичный инвентарь.

### 8. Регистрация и участие в глобальных онлайн-соревнованиях.

Сценарий регламентирует вход в турнирную сетку и обновление таблиц лидеров при финише.

* **8.1. Участник.** Авторизованный пользователь приложения.
* **8.2. Окружение.** В системе объявлен открытый международный челлендж или онлайн-марафон с призовым фондом в игровой валюте.
* **8.3. Основной сценарий.**
  1. Пользователь заходит во вкладку соревнований и нажимает кнопку участия в марафоне.
  2. Сервис геймификации добавляет идентификатор пользователя в турнирную таблицу этого события внутри реляционной базы данных.
  3. Пользователь выполняет беговую или силовую активность, телеметрия которой пишется по сценарию 1.
  4. При фиксации финиша домен телеметрии выбрасывает событие в шину. Сервис геймификации подхватывает его и обновляет позицию пользователя в таблице лидеров.
  5. Обновленная таблица лидеров кэшируется в оперативной памяти для мгновенного просмотра другими участниками.
  6. При завершении события в случае получения призового места пользователю начисляется локальная валюта и трофеи.
* **8.4. Альтернативный сценарий (Дисквалификация).**
  1. Если при проверке трека в домене телеметрии выявляются аномалии, событие финиша помечается как подозрительное.
  2. Результат не идет в общую таблицу лидеров, а отправляется на проверку модераторам во избежание жульничества.
